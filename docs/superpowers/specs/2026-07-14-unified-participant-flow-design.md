# 统一房间头像连接区设计

## 背景

房间页中，房间码与文件上传/接收状态区域之间的头像展示目前按角色采用了两种不同结构：

- 发送页显示发送者头像、连接状态与接收者头像组。
- 接收页先单独显示当前接收者身份，再嵌套一组“发送者到接收者”的连接流，导致本人头像重复、方向和间距不统一。

本次只统一该头像连接区，不改变房间权限、接收者选择、WebRTC 连接或文件传输协议。

## 目标

- 发送页和接收页使用同一套视觉组件与状态语义。
- 左侧始终是发送者，右侧始终是接收者组，不随当前设备角色翻转。
- 通过连接线本身表达等待、已连接、传输中、完成和错误状态。
- 保留发送端的接收者选择能力，接收端保持只读。
- 在桌面和窄屏上保持紧凑、清楚且可访问。

## 采用方案

继续使用并收敛现有 `TransferPeerFlow`，使其成为唯一的头像连接组件。`TransferPanel` 和 `ReceiverPanel` 继续负责各自的业务区域，但两者顶部都直接渲染相同结构的 `TransferPeerFlow`。接收页删除额外的当前接收者身份行，避免重复头像。

不把接收者选择弹窗和连接区整体提升到 `App`：该方案会扩大状态提升范围，并不能为本次视觉统一带来额外用户价值。

## 布局与头像规则

- 连接区占满内容栏宽度，发送者头像靠左，接收者头像组靠右，中间的状态轨道自适应填充剩余空间。
- 头像沿用现有基于 `avatarSeed` 的确定性样式和现有尺寸体系。
- 右侧有 1 至 5 名接收者时，展示全部头像，并使用小幅负间距形成紧凑头像组。
- 接收者超过 5 名时，展示前 4 个真实头像；第 5 个头像位改为总人数徽标，例如共有 12 名时显示 `12`，不显示 `+8`。
- 没有接收者时，右侧保留一个淡色圆形占位，内部使用 `person_add` 或等义的现有 Material Symbol，明确表达正在等待接收者。
- 窄屏缩小头像和轨道间距，但不改变左右角色顺序，不允许头像组挤出容器。

## 状态与动效

组件接收明确的视觉阶段，并将业务状态映射为以下表现：

| 阶段 | 轨道表现 | 图标 |
| --- | --- | --- |
| `connecting` / `requesting` | 三个紫色小点依次缩放和改变透明度 | 无 |
| `idle` | 低对比度静态实线 | 无 |
| `transferring` | 向右流动的紫色虚线 | 可使用小型传输图标，但不增加第二种强调色 |
| `complete` | 静态实线 | 小型 `check` |
| `error` | 中断样式的静态线 | 小型 `link_off` |

等待动画通过 `transform` 和 `opacity` 完成；传输动画通过轨道内伪元素或 SVG 虚线的位移完成，避免修改布局属性。所有动画都必须在 `prefers-reduced-motion: reduce` 下停用并保留可辨识的静态状态。

发送端没有已就绪接收者时显示 `connecting`；等待接收者确认文件时显示 `requesting`；文件正文传输时显示 `transferring`；传输完成或失败分别显示 `complete` 和 `error`。接收端根据发送者是否存在、点对点连接以及当前接收状态映射到相同阶段。

## 交互与可访问性

- 发送端存在可选择接收者且传输未锁定时，整个头像连接区继续作为接收者选择按钮。
- 接收端连接区为只读状态，不创建虚假按钮。
- `role="status"`、`aria-live="polite"` 和具体状态标签继续保留；装饰头像、轨道和图标不重复播报。
- 点击目标在可交互时保持至少 44 像素高，键盘焦点使用现有紫色焦点边框。
- 总人数徽标的可访问标签说明接收者总数；完整接收者名单仍通过现有选择弹窗提供，不把所有姓名塞入视觉轨道。

## 组件边界与数据流

- `TransferPeerFlow` 负责头像裁剪、人数徽标、占位头像、轨道状态、动画钩子和可访问状态。
- `TransferPanel` 负责把已连接/已选择接收者和发送活动阶段映射为连接区属性，并保留接收者选择弹窗。
- `ReceiverPanel` 负责把发送者、房间接收者和接收状态映射为同一连接区属性，并删除重复的本人身份行。
- `App` 继续提供现有房间成员、就绪连接和传输状态；不改变 API、共享协议或会话状态结构。

## 异常与边界

- 发送者尚不可用时，接收页仍保留布局位置并显示等待连接的无障碍文案，不渲染伪造发送者头像。
- 接收者列表动态减少时，头像组和总人数徽标立即按当前列表重算。
- 活动传输只展示该活动涉及的接收者；活动结束后恢复显示当前可用接收者。
- 接收者总数徽标支持两位及以上数字，使用 `tabular-nums`，不得撑破圆形容器。

## 验证

- 组件测试覆盖 0、1、5、6 名接收者，确认第 6 名触发“前 4 个头像 + 总人数徽标”。
- 组件测试覆盖等待点、静态实线、流动虚线、完成图标和错误图标的状态映射。
- 发送页测试确认连接区仍可通过键盘打开接收者选择弹窗，并保留已选择人数语义。
- 接收页测试确认当前接收者头像不再重复，且发送者始终在左、接收者组始终在右。
- 运行前端单元测试、类型检查、lint 和构建。
- 在真实浏览器中分别检查发送页与接收页的桌面和窄屏布局，以及减少动态效果模式。

## 非目标

- 不改变房间码区域、文件上传区或接收文件弹窗的业务行为。
- 不改变房间成员上限、接收者选择规则或传输协议。
- 不引入新图标库、动画库、颜色或阴影。
