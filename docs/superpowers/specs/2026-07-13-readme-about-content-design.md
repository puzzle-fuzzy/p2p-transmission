# README 与 About 内容完善设计

日期：2026-07-13

## 目标

让项目同时对三类读者清晰：

1. 想直接使用产品的普通用户。
2. 想运行、修改和测试项目的开发者。
3. 想在腾讯云单机上部署项目的维护者。

同时在 Web 产品内增加一个不打断传输流程的 About 入口，让用户在不阅读源码的情况下理解产品的工作方式、隐私边界和使用限制。

## 已确认的产品事实

文档与 About 只能描述当前实现已经保证的行为：

- 产品地址为 `https://p2p.yxswy.com`。
- 不需要注册账号，API 负责临时访客、房间、加入授权和 WebSocket 信令。
- 文本和文件内容通过浏览器 WebRTC DataChannel 传输，API 不保存或中继应用载荷。
- coturn 只在需要时中继加密的 WebRTC 流量，不能读取文本正文或文件内容。
- 6 位房间码是公开标识，不是成员授权凭证。
- 邀请链接包含加入权限，只能发送给可信接收者。
- 仅输入房间码会创建加入申请，需要房主批准，接收者还要完成最终加入。
- 房间默认有效期为 30 分钟。
- 一个文件批次最多 10 个文件，总大小最多 100 MiB。
- API 重启后 SQLite 恢复仍在生命周期内的房间和申请；在线 WebSocket 连接会断开并重新连接。
- 当前生产部署为单 API 实例，不承诺多实例状态同步。

避免使用“绝对安全”“服务器完全无法接触任何数据”等超出实现边界的表述。

## 方案选择

### 方案 A：分层文档 + 站内 About 弹窗（采用）

- 根目录 README 作为产品介绍、用户入口、开发入口和部署入口。
- `docs/user-guide.md` 承担普通用户的完整操作说明。
- `apps/web` 增加统一 About 弹窗，覆盖大厅和房间页面。
- `apps/web/README.md`、`services/api/README.md`、`deploy/README.md` 保留技术细节并互相链接。

优点是产品入口和工程入口都清晰，而且不需要引入路由系统或新的页面生命周期。About 弹窗能在用户当前操作上下文中提供说明，不会把用户带离房间。

### 方案 B：README 只做产品首页

README 只保留产品介绍和快速开始，所有开发、测试、部署内容拆到 docs。

优点是首页更短；缺点是开发者需要额外跳转，当前仓库的部署和安全协议说明会变得分散。

### 方案 C：新增独立 About 页面

增加 `/about` 页面或独立 landing page，并在应用内通过路由跳转。

优点是可承载更丰富的品牌内容；缺点是需要引入路由、页面加载和返回行为，超出本次内容完善的必要范围。

## 文档设计

### 根目录 README

采用以下顺序：

1. 产品一句话定位、生产地址和核心能力。
2. 30 秒快速使用：创建房间、分享邀请、传输文本或文件。
3. 隐私与安全模型：WebRTC DataChannel、API 职责、TURN 边界、邀请链接敏感性。
4. 使用限制：房间时限、文件批次限制、浏览器和网络要求。
5. 项目结构与数据流：Web、API、SQLite、WebSocket、coturn。
6. 本地开发和验证命令。
7. 腾讯云生产部署入口和关键安全组说明。
8. 故障排查：服务器连接、房间过期、直连失败、TURN 失败、部署后旧页面。
9. 相关文档索引和当前单机边界。

README 不虚构许可证、贡献流程、Issue 地址或第三方服务承诺；仓库没有相应事实时不添加这些段落。

### `docs/user-guide.md`

以普通用户任务为中心，不使用 API 或 WebRTC 术语作为前置要求：

- 创建房间并复制房间码。
- 通过安全邀请链接加入。
- 仅通过房间码申请加入并等待房主批准。
- 房主批准、接收者完成加入。
- 发送文本和文件，理解等待、连接、传输和失败状态。
- 房间过期、浏览器关闭、网络切换和 TURN 中继的处理建议。
- 隐私提醒：不要把包含加入权限的邀请链接发给不可信的人。

每个失败场景都说明“发生了什么”和“下一步做什么”，不只展示错误名称。

### 既有技术 README

只做一致性整理，不重写已经准确的协议细节：

- 统一产品名称、术语、生产域名和 Bun 版本。
- 增加根 README 和用户指南的链接。
- 保留开发、测试、部署和 TURN 验收命令。
- 明确 SQLite 持久化范围和单实例限制。

## About 交互设计

### 入口

- 大厅底部提供文字按钮“关于 P2P Transmission”。
- 房间顶部提供 info 图标按钮，`aria-label` 和 `title` 使用“关于 P2P Transmission”。
- 两个入口打开同一个 `AboutDialog`，不复制两套文案。

### 弹窗结构

使用项目现有的原生 `<dialog>` 模式和深色设计 tokens：

1. 标题：`关于 P2P Transmission`。
2. 一句话：`不注册，不上传，直接把内容传给对方。`
3. “它是怎么工作的”：三步说明创建房间、确认加入、浏览器直连或通过 TURN 中继。
4. “隐私与安全”：说明 DataChannel、API、TURN 和邀请链接边界。
5. “使用前知道”：30 分钟房间、10 个文件/100 MiB 批次、网络兼容性和重要文件备份提醒。
6. “构建信息”：生产地址和构建版本；未注入版本时显示“开发构建”，不伪造版本号。
7. 底部按钮“关闭”。

弹窗内容可滚动但默认保持短小；不在 About 中放置难以验证的法律声明或营销夸张语句。

### 可访问性和状态

- 使用 `aria-labelledby` 关联标题。
- 打开后聚焦关闭按钮，支持 Escape 关闭和点击明确关闭按钮。
- 使用现有的 `prefers-reduced-motion` 约束，不添加必要的动画依赖。
- 文案使用普通中文，按钮名称描述实际动作。
- About 不读取 visitor token、房间邀请 token 或其他敏感状态。

### 构建版本

前端使用 `VITE_APP_VERSION` 注入构建版本；没有注入时显示“开发构建”。生产部署通过 `.env` 或 Docker build arg 注入明确版本，版本号不能来自用户可控输入。

## 实现边界

本次不做：

- 新增路由系统或独立 About 页面。
- 新增登录、账号、历史记录或云端文件存储说明。
- 修改 WebRTC、SQLite、TURN 或房间授权协议。
- 添加未经确认的许可证、隐私政策法律文本或客服承诺。
- 大规模视觉重做；About 沿用现有深色、紫色强调和琥珀色文字系统。

## 验证方案

- 新增 About 组件测试：标题、关键文案、关闭行为、键盘语义和版本 fallback。
- 更新 App 测试：大厅和房间入口都能打开同一个 About 状态。
- 增加文档链接检查，确保 README 指向的仓库内文件存在。
- 运行 `bun run verify -- --force`。
- 运行 `bun run e2e`，确认 About 不影响建房、加入、文本和文件传输。
- 手动打开生产地址确认 About 文案与当前部署事实一致。
