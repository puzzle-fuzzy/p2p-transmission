# 统一上传与文本项复制设计

日期：2026-07-14

## 背景

当前发送端在 `TransferPanel` 中通过“传输文本 / 传输文件”两个 Tab 区分两套流程：文本调用 `PeerSession.offerText`，文件调用 `PeerSession.offerFiles`。这导致发送入口、校验、活动状态、重试逻辑和接收端展示都存在分支。

新的产品模型是：发送端只有一个上传区域。用户可以选择、拖拽或粘贴内容；粘贴的真实文件按文件项目处理，粘贴的纯文本按文本项目加入同一个传输列表。所有新发送内容统一走文件传输协议。

## 目标

- 移除发送端的文本/文件 Tab 和文本输入框。
- 通过一个上传区域统一处理文件选择、拖拽和剪贴板粘贴。
- 仅在上传区域获得焦点或被点击后识别粘贴，避免干扰房间码、弹窗和其他输入框。
- 粘贴文件或文本前弹出确认，不在确认后自动发送。
- 文本作为 UTF-8 的 `.txt` 传输项目进入现有文件队列，并复用文件数量、大小和批次校验。
- 接收方继续使用统一的文件请求和接收流程。
- 接收完成后，`.txt` / `text/plain` 项显示明确的“复制内容”操作；复制成功反馈需要保持可感知，同时保留下载能力。
- 删除旧版本的文本传输协议、文本接收状态和文本接收弹窗；新版本只保留文件传输协议。

## 已确认的交互决策

### 1. 统一上传区域

上传区域是发送端唯一的内容入口，提示文案应覆盖三种方式：选择文件、拖拽文件、粘贴内容。文件列表继续使用现有 `FileSelection`，发送按钮显示“发送 N 项”或等价文案，不把文本项目单独称为一种传输模式。

上传区域需要具备可聚焦语义，例如 `tabIndex=0` 和明确的键盘焦点样式。粘贴监听绑定到该区域，不注册页面级全局 `paste` 监听。

### 2. 粘贴识别与确认

处理 `ClipboardEvent` 时优先检查 `clipboardData.files`：

- 有一个或多个真实文件时，将这些文件作为候选项目展示在确认弹窗中。
- 没有真实文件但存在 `text/plain` 时，将纯文本作为一个文本项目展示在确认弹窗中。
- 剪贴板同时包含文件和文本表示时，只处理真实文件，避免截图或富文本剪贴板产生重复文本项目。
- 无文件且无纯文本时，不拦截也不打开弹窗。

确认弹窗的行为：

- 文件粘贴：显示文件名、数量和总大小。
- 文本粘贴：显示截断预览、字符数，并明确说明会作为 `粘贴内容.txt` 添加。
- “取消”不改变当前文件列表和剪贴板内容。
- “添加到传输列表”只添加项目，用户仍需主动点击发送。
- 粘贴项目通过现有文件选择校验；超过文件数量、单文件大小或批次大小时，在确认或添加阶段显示稳定的错误信息。

### 3. 文本项目的内部表示

产品层将其称为“文本项目”，协议层将其视为普通文件项目：

```ts
new File([plainText], '粘贴内容.txt', {
  type: 'text/plain',
  lastModified: Date.now(),
})
```

文本按浏览器提供的 `text/plain` 原样保存，包括换行、空格和 Unicode 字符。使用现有 `addFileSelections` 加入 `FileSelection`，不增加第二套发送项目模型，也不再使用旧文本传输的字符数限制。

同一列表中出现多个文本项目时，文件名需要稳定去重，例如 `粘贴内容.txt`、`粘贴内容 (2).txt`。文本项目和普通 `.txt` 文件在传输层保持一致；是否显示“文本”辅助标识由文件名和 MIME 类型决定。

### 4. 发送、重试与接收

`App` 的新发送状态只保留文件 payload：

```ts
{ kind: 'file'; selections: FileSelection[]; peerIds: string[] }
```

发送和重试统一调用 `peerSession.offerFiles`。`TransferPanel` 不再接收 `onSendText`，也不再维护文本发送状态。发送活动、文件进度、失败和重试继续沿用现有文件流程。

接收方对新版本发送的文本项目不增加独立文本请求流程，仍使用 `IncomingFileRequestDialog`。文件请求列表可以通过 `.txt` 后缀或 `text/plain` 类型识别文本项目，但不改变接受、拒绝、取消和传输状态机。

由于产品尚未大规模使用，本次直接删除 `transfer:text`、文本 receipt、`offerText`、文本接收队列、`planIncomingText` 和 `ReceivedTextDialog`。这样协议、状态机、发送 UI 和接收 UI 只有一套文件传输模型，不保留已经废弃的分支。

### 5. 接收完成后的 TXT 复制

接收文件完成后，符合以下任一条件的项目显示“复制内容”按钮：

- 文件名以 `.txt` 结尾（不区分大小写）。
- 文件 MIME 类型为 `text/plain`，如果该类型在接收文件元数据中可用。

点击“复制内容”后：

1. 从已接收的 Blob/File 读取文本内容。
2. 通过 `navigator.clipboard.writeText` 写入剪贴板。
3. 成功后将按钮状态改为“已复制”，保持约 2 秒，再恢复“复制内容”。
4. 失败时显示可理解的错误提示，并保留“下载”操作。

复制操作必须是明确按钮，不点击整行或文件名就执行，避免误触。下载按钮继续保留，文本内容较大或浏览器不允许剪贴板写入时，用户仍可以下载文件。

复制状态应按 `fileId` 管理，避免多个文本项目之间相互闪烁或错误显示成功状态；组件卸载或关闭弹窗时清理相关定时器。

## 实现范围

预计修改以下区域：

- `apps/web/src/components/TransferPanel.tsx`：移除 Tab 和文本发送 UI，增加统一上传区域的粘贴事件和候选确认入口。
- 新增粘贴确认弹窗组件及单元测试。
- `apps/web/src/App.tsx`：移除新发送的文本 payload、文本发送处理和重试分支；接入粘贴项目；增加接收 TXT 复制回调或 Blob 读取能力。
- `apps/web/src/features/transfer/file-selection.ts`：增加文本项目的 File 创建、文件名去重或相关测试辅助逻辑。
- `apps/web/src/components/IncomingFileRequestDialog.tsx`：在接收完成的文本项目上显示“复制内容”，保持下载按钮和持久化成功反馈。
- `packages/contracts/src/transfer.ts`：移除旧文本控制帧、文本 receipt 和文本字符数限制。
- `apps/web/src/features/transfer/peer-session.ts`：移除文本 offer、文本接收事件、文本 receipt 和文本专用状态。
- `apps/web/src/features/transfer/ui-state.ts`：移除文本 outgoing activity 和文本接收队列规划器。
- 删除 `apps/web/src/components/ReceivedTextDialog.tsx` 及其测试。
- 相关 `TransferPanel`、`App`、`IncomingFileRequestDialog`、文件选择和 `peer-session` 测试。
- `apps/web/e2e/room-transfer.spec.ts`：增加真实浏览器粘贴确认、取消、文本项目按文件协议传输、接收端复制内容和上传区域焦点范围测试。

本次直接删除 `transfer:text`，不保留旧客户端兼容路径；产品尚未大规模使用，统一协议模型的长期维护成本更低。

## 测试验收标准

- 页面不再渲染“传输文本 / 传输文件”Tab。
- 文件选择和拖拽行为保持可用。
- 上传区域聚焦后粘贴文件会弹出确认；取消不会改变列表，确认会加入列表。
- 上传区域聚焦后粘贴纯文本会弹出预览确认，并加入一个 `.txt` 项目。
- 上传区域外的房间码输入框和其他弹窗粘贴行为不被拦截。
- 混合剪贴板只产生真实文件项目，不产生重复文本项目。
- 文本项目发送调用 `offerFiles`，不会调用 `offerText`；重试也保持相同语义。
- 接收端接受文本项目后显示在文件接收流程中。
- 接收完成后，点击明确的“复制内容”可以复制完整文本；成功状态至少保持可感知时间，且下载仍可用。
- 剪贴板写入失败、文本过大或缺少 Clipboard API 时有稳定错误提示。
- 协议解析器拒绝旧 `transfer:text`，所有可接受的内容都必须是文件请求。
- `bun run verify` 和真实浏览器 E2E 全部通过。

## 风险与回滚

主要风险是浏览器对剪贴板文件、权限和 `navigator.clipboard.writeText` 的支持差异。功能必须提供清晰的取消、失败和下载回退路径，E2E 测试使用真实浏览器上下文验证授权和焦点行为。

如果需要回滚，可恢复发送端文本 Tab、`offerText` 和旧文本接收流程；本次删除的协议分支与接收 UI 不应通过新增兼容代码重新保留。
